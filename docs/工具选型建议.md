# 工具选型建议

本节基于 [SDD 落地实践指南](../README.md) 的核心理念，对常见规范驱动开发工具进行对比，并给出选型建议，陈述各工具在**以该指南为基准在实际开发项目中落地**的能力。

---

## 一、指南对落地工具的核心要求（摘要）

| 维度               | 指南要求                                                                                                               |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **生命周期**       | 明确区分：Specify（规格定义）→ Plan（方案规划）→ Tasks（任务拆解）→ Implement → Deliver → Iterate                      |
| **规范形态**       | 按功能/模块的规格文件，存放在 `specs/` 或 `docs/sdd/`，与代码同库；包含规格 ID、业务目标、范围、输入输出、**验收条件** |
| **验收标准**       | 每条验收写成「给定 X，系统应 Y，可观测/可验证为 Z」，并能与测试用例一一对应                                            |
| **规格与设计分离** | 规格只写「做什么、验收」；「怎么做」放在 Plan/设计文档中，并**引用规格 ID**                                            |
| **全角色协同**     | BA 主导起草，开发/测试参与补充；规格评审通过后冻结，作为法定依据                                                       |
| **多团队**         | 唯一规格 ID、规格共享、接口在规格中以契约形式写清                                                                      |
| **与现有流程衔接** | 不推翻 Word/会议，而是把「结构化规格」作为澄清会议的**输出**，逐步成为唯一依据                                         |
| **工具定位**       | 轻量、规范即数据可解析、可与任务/PR 关联、可扩展门禁（如 validate / trace）                                            |

---

## 二、各工具落地能力陈述

### 2.1 Spec-Kit（GitHub）

- **工作流**：Constitution（项目原则）→ Specify → Plan → Tasks → Implement，与指南的 Specify / Plan / Tasks / Implement 一一对应。
- **规范形态**：`specs/NNN-feature-name/` 下 `spec.md`（需求）、`plan.md`（技术设计）、`tasks.md`（任务），Markdown、与代码同库。
- **规格与设计分离**：spec 只写业务与验收，plan 写技术方案并引用规格，符合指南「规格只写做什么、设计写怎么做并引用 SPEC-ID」。
- **全角色共识**：Constitution 定义项目级原则，对应指南的「全角色协同共识」。
- **轻量与扩展**：CLI + 脚本，无强制外部服务，可在此基础上做 `sdd validate`、`sdd trace` 等。
- **多团队**：通过目录与命名约定即可实现 SPEC-ID、跨团队共享同一份 specs。
- **与 BA 流程衔接**：澄清会议产出的「结构化规格」可直接写成 `spec.md`，不改变现有 Word + 会议形式，只增加「规格产出」出口。

**落地能力结论**：阶段完整、规格/设计分离、Constitution、轻量、易与「BA Word + 澄清 → 结构化规格」衔接，**与指南的契合度最高**，最适合作为以该指南为基准在实际项目中落地的主工具。

---

### 2.2 OpenSpec（Fission-AI / openspec.dev）

- **工作流**：Propose（Delta Spec）→ Implement → Archive，偏「变更驱动」：先改规格、再实现、再归档到主规格。
- **规范形态**：按能力组织的 `spec.md`（如 `openspec/specs/auth-session/spec.md`），常用 Gherkin 风格（Requirement + Given/When/Then），利于可验证验收。
- **优点**：Delta 评审清晰、验收可执行、多 AI 支持、无 API Key、规格在仓库内。
- **与指南的差距**：无显式 Plan、Tasks 阶段；更偏单系统/API 契约；对「BA 主导、多角色评审、多团队共享 SPEC-ID」需在流程上自行补齐；与「Word + 需求澄清 → 结构化规格」的衔接不如 Spec-Kit 直观。

**落地能力结论**：适合做**接口/契约层**或**单能力规格**的落地，或在已有 Spec-Kit 基础上局部采用（如用 Gherkin 写验收）；**单独作为以该指南为基准的全流程落地工具，不如 Spec-Kit 贴合一整套指南**。

---

### 2.3 Speck（基于 Spec-Kit）

- 与 Spec-Kit 同源、同方法论，文件格式兼容（`specs/` 下 spec / plan / tasks）。
- 在 Spec-Kit 基础上增加对 Claude Code 的深度集成（多仓库、Worktree、Slash 命令等）。

**落地能力结论**：若团队主用 **Claude Code**，用 Speck 能更好践行同一套指南；若主用 **Cursor** 或其他 AI，用 **Spec-Kit** 即可，指南落地能力一致，只是集成对象不同。

---

### 2.4 SpecPulse

- 公开信息较少，文档与社区不足以系统对照「六阶段、规格/设计分离、多角色、多团队、与 BA 流程衔接」等要求。

**落地能力结论**：目前**无法判断**能否以该指南为基准在实际项目中完整落地，暂不推荐作为主选工具。

---

## 三、选型建议汇总

| 工具          | 落地能力（以本指南为基准）                                        | 建议                                           |
| ------------- | ----------------------------------------------------------------- | ---------------------------------------------- |
| **Spec-Kit**  | 高：阶段完整、规格/设计分离、Constitution、轻量、易与 BA 流程衔接 | **首选**，作为全流程落地主工具                 |
| **OpenSpec**  | 中高：验收与契约强，缺 Plan/Tasks 显式阶段及多角色/多团队流程     | **补充**，用于接口/契约或验收格式参考          |
| **Speck**     | 与 Spec-Kit 相当，针对 Claude Code 优化                           | 主用 Claude Code 时可选 Speck，否则用 Spec-Kit |
| **SpecPulse** | 待评估：公开信息不足                                              | 暂不作为主选，待资料充分后再评估               |

---

## 四、结论

以本指南为基准在实际开发项目中落地时，**优先采用 Spec-Kit**（或主用 Claude Code 时采用 Speck）；需要强契约与可执行验收时，可结合 OpenSpec 的 Delta 与 Gherkin 风格作为补充。
